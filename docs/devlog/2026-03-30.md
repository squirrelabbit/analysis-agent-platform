# 2026-03-30

## 오늘의 질문

- Python MVP에 섞여 있던 API, workflow, AI worker 책임을 어떤 런타임 경계로 다시 나눌지 정리했다.
- `request -> plan -> execute` 흐름을 운영형 구조로 옮길 때 어떤 컴포넌트를 먼저 세워야 하는지 결정할 필요가 있었다.

## 왜 이게 문제인가

- API, 실행 상태, LLM 호출, 데이터 계산이 한 덩어리로 남아 있으면 Skill 수가 늘수록 재시도와 장애 대응이 어려워진다.
- 구조를 먼저 나누지 않으면 이후 `dataset build`, `result snapshot`, `scenario`, `final_answer` 같은 기능이 모두 같은 레이어에 엉키게 된다.

## 시도한 것

- `Go control plane`, `Temporal runtime`, `Python AI worker`를 별도 디렉터리와 entrypoint로 세웠다.
- `apps/control-plane/internal/http/server.go`, `internal/service/*`, `internal/workflows/*` 기준으로 API와 workflow 경계를 잡았다.
- `workers/python-ai/src/python_ai_worker/main.py`, `tasks.py`를 중심으로 worker 진입점과 task routing을 만들었다.
- 개발용 compose, smoke script, OpenAPI/Swagger까지 같이 넣어 초기 구조를 바로 검증할 수 있게 했다.

## 관찰한 점

- 사실: 당일 핵심 커밋은 `88c3f89` `add go control plane and temporal execution runtime`, `dc54b37` `add python ai worker and local analysis stack`, `bbef6a5` `add openapi spec and swagger ui`였다.

- 사실: 초기 문서와 구조도도 함께 추가돼 이후 작업이 코드와 문서를 같이 따라가도록 출발점을 맞췄다.

## 현재 판단

- 이 날의 핵심은 기능 추가보다 `운영형 런타임 경계 확정`이었다.
- 이후 구현은 대부분 이 구조 위에서 `dataset build`, `execution result`, `scenario`, `profile`을 덧붙이는 방식으로 전개됐다.

## 남은 리스크

- 확인 필요: 이 시점에는 아직 shared skill metadata, 비정형 저장 계약, startup reconciliation 같은 운영 장치가 없었다.
- 확인 필요: Rust worker는 스캐폴드만 있고 실제 hot path 분리 기준은 아직 없었다.
