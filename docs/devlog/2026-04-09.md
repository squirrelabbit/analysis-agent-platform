## 오늘의 질문
- 분석팀이 실행 중간중간 결과를 보면서 어디가 잘됐고 어디가 잘못됐는지 확인하려면, 현재 백엔드가 어느 정도까지 상태를 보여줄 수 있는가?
- `플래닝 결과`, `실행 상태`, `실행 결과`는 이미 있는데, `실행 중 live progress`와 `partial result`를 추가하려면 어디를 바꿔야 하는가?

## 왜 이게 문제인가
- 기존 구조는 `execution completed` 이후에 `result_v1`, `final_answer`, `step_hooks`가 한 번에 저장되는 흐름이었다.
- 그래서 프론트는 `queued/running/waiting/completed/failed` 정도는 볼 수 있어도, 지금 어떤 step이 돌고 있는지와 실행 도중 나온 중간 산출물을 확인하기 어려웠다.
- 분석팀이 step별 중간 결과를 보고 판단하려면, 실행 종료 후 snapshot이 아니라 실행 중에 step event와 partial artifact가 쌓여야 한다.

## 관찰한 점
- 현재 구조에서는 planning이 동기라 `플래닝중` 상태를 별도 백엔드 상태 머신으로 만들지 않아도, 프론트 로컬 로딩으로 당장 대응 가능하다.
- 반면 execution은 live progress가 없으면 분석팀 요구를 충족시키기 어렵다. `STEP_*` 이벤트와 partial artifact 저장이 먼저 있어야 프론트가 중간 결과 카드를 만들 수 있다.
- 기존 `step_hooks`는 `WORKFLOW_COMPLETED` payload에만 붙었는데, progress hook으로 execution 이벤트와 artifact 저장을 분리하니 완료 전에도 중간 상태를 재구성할 수 있었다.

## 현재 판단
- 오늘 기준으로 프론트가 바로 쓸 수 있는 최소 백엔드 범위는 준비됐다.
- 새로 추가한 경로:
  - `GET /projects/{project_id}/executions/{execution_id}/progress`
- 이 응답에서 제공하는 것:
  - 전체 실행 상태
  - step별 상태
  - 현재 running step
  - waiting 상태
  - 현재까지 저장된 artifact 목록
  - `result_preview`
- 즉 `플래닝 결과 -> 실행 상태 -> 실행 중 partial preview -> 실행 결과`까지의 흐름은 이제 API 기준으로 연결 가능하다.

## 남은 리스크
- 확인 필요: planning은 여전히 동기라, backend-level `planning` 상태가 꼭 필요해지면 별도 async planning job이 필요할 수 있다.
- 확인 필요: 현재 progress는 step 단위 상태와 partial preview까지는 제공하지만, 별도 `events` 조회 API는 아직 없다.
- 확인 필요: step 완료 시 artifact를 즉시 저장하므로, 매우 큰 artifact가 많은 시나리오에서는 execution row 저장 부담을 추가로 봐야 할 수 있다.
