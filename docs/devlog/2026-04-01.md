# 2026-04-01

## 오늘의 질문

- 비정형 dataset build와 retrieval 경로를 `JSONL sidecar` 중심에서 `Parquet + local embedding + pgvector` 중심으로 얼마나 옮길지 정리했다.
- execution 결과를 더 재현 가능하게 만들기 위해 usage, step hook, rule config를 어디까지 metadata로 남길지 검토했다.

## 왜 이게 문제인가

- `prepare -> sentiment -> embedding -> evidence`가 파일 경로와 JSONL fallback에 많이 기대면 저장 계약과 메모리 사용이 불안정해진다.
- dense embedding과 retrieval이 들어와도 운영 메타데이터가 약하면 어느 model, 어느 규칙, 어느 artifact로 실행됐는지 추적하기 어렵다.

## 시도한 것

- `prepared.parquet`, `chunks.parquet`, `embeddings.index.parquet`를 중심으로 저장 경로를 옮기는 체크포인트를 추가했다.
- `pgvector`를 embedding clustering과 retrieval에서 우선 쓰도록 바꿨다.
- 로컬 임베딩 평가 fixture와 Postgres reset helper를 추가해 dense embedding 경로를 반복 검증할 기반을 만들었다.
- support skill sidecar artifact 저장, evidence prompt context compaction, usage/cost metadata, runtime step hook, layered rule config를 함께 넣었다.
