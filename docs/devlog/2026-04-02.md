# 2026-04-02

## 오늘의 질문

- execution 결과를 사용자에게 보여줄 표현 레이어를 어디까지 control plane 안에서 정리할지 검토했다.
- 반복 질문을 strict 시나리오로 등록하고, `시나리오 -> plan -> execute` 경로를 별도 API로 만들지 정리했다.

## 왜 이게 문제인가

- raw artifact만으로는 execution 목록, 결과 비교, 보고서 초안 같은 후속 기능을 일관되게 만들기 어렵다.
- 반복 질문이 늘어날수록 자유 질문 planner만으로는 재현성이 떨어지고, 운영자가 대표 질문을 템플릿처럼 관리하기 어렵다.

## 시도한 것

- `result_v1` 표현 레이어와 execution snapshot 저장을 추가했다.
- execution 목록 preview와 report draft API를 만들어 snapshot 재사용 경로를 붙였다.
- 시나리오 등록, 상세 조회, strict plan 생성, 일괄 import, one-shot execute API를 순서대로 추가했다.
- 축제용 strict 템플릿 fixture를 넣어 시나리오 import와 실행 예시를 문서화했다.

## 관찰한 점

## 현재 판단

- 이 날의 핵심은 `결과 표현층`과 `strict 시나리오 시스템`을 한 번에 세운 것이다.
- 자유 질문과 시나리오 질문이 모두 결국 execution snapshot으로 합류하게 되면서, 이후 `final_answer`나 프론트 결과 화면을 붙일 바탕이 생겼다.

## 남은 리스크

- 확인 필요: strict 시나리오는 재현성은 높지만 질문-데이터 정합성이 어긋나면 품질이 급격히 떨어진다.
