- execution이 이미 `completed`인데 응답의 diagnostics에 예전 `waiting_for=embeddings` 같은 정보가 남으면 운영 화면과 사용자 해석이 어긋난다.

## 관찰한 점
- full-dataset `embedding_cluster`는 precomputed cluster artifact를 읽고, subset/filter가 앞에 붙는 경우만 기존 on-demand 경로를 유지하는 방식이 현재 repo 구조와 가장 충돌이 적다.
- membership parquet를 추가하니 `issue_cluster_summary`에서 sample 수를 유연하게 보강할 수 있었고, `issue_evidence_summary`도 lexical fallback이 아니라 `cluster_membership` 기반 근거 선택으로 내려갈 수 있었다.
- cluster smoke에서 `selection_source=cluster_membership`, `citation_mode=chunk`, `cluster_membership_ref` 전파를 확인했다.
- stale waiting 정보는 runtime이 아니라 result/diagnostics 조립부에서 남고 있었다. 실행 상태를 기준으로 diagnostics를 자르는 편이 영향 범위가 작았다.

## 현재 판단
- 오늘 기준 cluster 구조는 1차 마감 가능한 수준이다.
- 확정한 방향:
  - `embedding_cluster`는 full-dataset 경로에서 precomputed cluster를 우선 사용한다.
  - cluster 산출물은 `summary JSON + membership parquet`로 분리 저장한다.
  - cluster membership은 drill-down API와 evidence/sample 보강에 실제로 사용한다.
  - 완료된 execution 응답에는 stale waiting diagnostics를 남기지 않는다.
- 즉, 오늘 작업으로 cluster 관련 핵심 흐름은 다음까지 닫혔다.
  - `prepare -> embedding -> cluster build -> auto resume -> cluster summary/evidence`

## 남은 리스크
- 확인 필요: subset/filter가 있는 cluster 질문은 여전히 on-demand fallback 경로에 의존하므로 큰 데이터셋에서 성능 병목이 다시 나타날 수 있다.
- 확인 필요: `cluster_membership.parquet`를 이후 프론트 drill-down에서 자주 읽게 되면 sidecar parquet만으로 충분한지, DB 승격이 필요한지 검토가 필요하다.
- 확인 필요: cluster label 품질 자체는 현재 `top_terms` 기반이라, 이후 실제 도메인 데이터에서 label 전략을 더 보강해야 할 수 있다.
