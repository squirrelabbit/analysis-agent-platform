## 오늘의 질문
- `embedding_cluster`를 현재처럼 execution 시점에 계산하는 방식으로 계속 가져갈지, 아니면 dataset-level 비동기 materialization 작업으로 바꿔야 하는지 검토했다.
- 현재 repo 안에서 함수 최적화만 하는 것이 최선인지, 구조를 바꾸는 것이 더 맞는지 판단이 필요했다.

## 왜 이게 문제인가
- 현재 `embedding_cluster`는 실행 시점에 `pgvector` row를 읽고, 다시 chunk parquet를 조회한 뒤, 전체 record를 메모리에 올려 clustering을 수행한다.
- 이 구조는 dataset 크기가 커질수록 IO, 메모리, 처리 시간이 같이 커진다.
- 같은 dataset version에 대해 cluster 결과를 반복적으로 조회할 가능성이 높은데, 매번 다시 계산하면 비용이 크고 실행 일관성도 떨어진다.

## 시도한 것
- 현재 구현 경로를 다시 읽고 병목 위치를 정리했다.
- execution-time 최적화 방향과 구조 변경 방향을 분리해서 검토했다.
- `dataset build job` 체계 위에 `cluster build`를 올리는 방식이 가능한지 확인했다.
- `cluster`를 새 dataset build type으로 추가하고, control plane / Temporal / worker를 모두 연결했다.
- `embedding_cluster`가 full-dataset 경로에서는 precomputed cluster artifact를 우선 읽고, subset/prior filter가 있을 때만 기존 on-demand 계산으로 fallback 하도록 1차 구현했다.

## 현재 판단
- 현재 repo 기준 단기 최적화는 여전히 필요하다.
  - SQL pushdown
  - parquet 재읽기 축소
  - 중간 record/member payload 축소
- 하지만 구조적으로는 `embedding_cluster`를 execution-time 계산에서 빼고, `dataset-level cluster build`로 materialize 하는 방향이 더 맞다.
- 1차 제품 규칙은 `dataset version 전체 기준 cluster`만 지원하는 것이 현실적이다.
- subset cluster는 당장은 미지원 또는 작은 subset에 대해서만 on-demand 재계산하는 방향이 적절하다.

## 남은 리스크
- 1차 구현은 `cluster_membership` 별도 저장 없이 summary JSON artifact만 materialize 한다. cluster membership 질의가 많아지면 `parquet sidecar` 또는 별도 table이 추가로 필요하다.
- 확인 필요: 현재 hybrid clustering 알고리즘을 그대로 build 단계로 옮겼다. `HDBSCAN` 또는 `MiniBatchKMeans` 계열로 바꿀지는 별도 검토가 필요하다.
- 확인 필요: `prepare/profile/embedding` 변경 시 cluster stale 판정 규칙을 어디까지 둘지 정하지 않았다.
- subset/prior artifact가 있는 `embedding_cluster`는 아직 기존 on-demand 계산 경로를 그대로 사용한다.
