# 2026-04-06

## 오늘의 질문

- dataset version마다 다른 prompt/rule/model recipe를 어디에 붙일지 정리했다.
- `prepare -> sentiment -> embedding -> cluster` build를 수동 호출이 아니라 운영형 비동기 orchestration으로 바꿀 수 있는지 검토했다.
- control plane 재시작이나 build 지연이 있어도 execution이 복구 가능한지 확인할 필요가 있었다.

## 왜 이게 문제인가

- dataset마다 전처리와 감성 프롬프트가 달라질 수 있는데 기본값이 코드에 흩어져 있으면 운영이 금방 꼬인다.
- build dependency를 사용자가 직접 `prepare -> resume`로 맞춰야 하면 execution UX와 운영 신뢰성이 떨어진다.
- Temporal dev persistence에 완전히 기대기 어려운 상태에서, 앱 자체의 복구 전략이 없으면 미완료 build와 execution이 유실될 수 있다.

## 시도한 것

- dataset version profile v1을 도입하고, `dataset_profiles.json`과 Markdown prompt template를 registry로 연결했다.
- `prepare`, `sentiment`, `embedding`을 async build job으로 빼고 Temporal build workflow로 이관했다.
- `prepare=eager`, `sentiment / embedding=lazy` 정책과 auto resume 체인을 붙였다.
- build workflow timeout/retry, queue 분리, concurrency 운영값을 고정했다.
- startup reconciliation을 추가해 재시작 시 build job과 execution을 다시 평가하도록 했다.

## 현재 판단

- 이 날의 핵심은 `질문 실행 엔진`을 `dataset build orchestration`과 분리해 운영형 구조로 올린 것이다.
- Temporal persistence가 완전하지 않아도 startup reconciliation과 build metadata가 있으면 앱 수준 recoverability를 어느 정도 확보할 수 있다는 판단이 섰다.

## 남은 리스크

- 확인 필요: Temporal workflow history 장기 보존은 아직 dev server 기본값을 따른다.
- 확인 필요: 실제 운영 환경에서 build queue concurrency 상한은 머신 자원 기준으로 다시 조정해야 한다.
- 확인 필요: cluster dependency는 이후 materialization 경로로 한 번 더 손볼 필요가 있었다.
